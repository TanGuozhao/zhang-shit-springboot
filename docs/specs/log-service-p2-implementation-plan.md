# Log Service P2 整体落地规划

**作者**: Codex  
**日期**: 2026-06-07  
**状态**: 已落地到当前仓库的实施规划  
**关联文档**: `docs/specs/log-service-p2-spec.md`、`docs/contracts/log-service-contract.md`、`可复用访问日志服务设计-2024091602016-谭国照.md`

## 1. 目标定位

本规划不是再做一个 MVP，而是把原始《可复用访问日志服务设计》收敛成当前仓库里真正可以推进上线的 P2 落地方案。目标是先把 `log-service` 建成一个独立、可运行、可演进的访问日志后端，完成接入、缓冲、写入、检索、指标、告警、导出、生命周期治理和运行态观测的生产可演进基线。

本轮遵循两个硬约束：

- 对外公开接口继续遵守 `docs/contracts/log-service-contract.md`，不破坏 `topbiz` 已预留的调用契约。
- 原文档中明显不适合当前仓库直接落地的“大而全平台能力”，拆分为“当前实现必须落地”和“后续阶段扩展”两层，而不是在一次提交里假装全部完成。

## 2. 与原文档的对齐与纠偏

当前实现已按以下原则对原文档做纠偏：

- 原文中的部分路径写成了 `/api/log/*`，当前仓库冻结契约是 `/api/logs/*`，已统一按冻结契约实现。
- 原文把“采集端配置、解析规则平台、资源指标平台、全量权限体系”都放进一个服务，这超出了当前多模块仓库的真实边界。当前实现将 `log-service` 聚焦为“访问日志接收与分析服务”，不承担外部采集代理编排。
- 原文中“导出大文件自动分片打包、站内信通知、跨服务权限过滤”属于平台级二期能力，本轮先落地异步导出任务、下载路径、脱敏、过期清理和运行态可观测。
- 原文要求 ClickHouse 按天分区、TTL 保留期清理，这部分已经纳入表结构与配置，并在实现上补充了失败批次归档目录。
- 原文要求查询有时间窗、关键词长度、分页限制、异常输入校验，本轮已补成服务级硬护栏。

## 3. 当前落地架构

### 3.1 数据面

写入路径：

1. 业务方调用 `POST /api/logs/ingest`
2. 接口完成基础字段校验、级别校验、时间范围校验
3. 日志先进入内存缓冲队列
4. 后台调度按批次或时间窗口刷盘
5. 优先写 ClickHouse，失败时进入重试仓
6. 超过最大重试次数后落本地失败批次目录，供运维排查

查询路径：

1. 公开查询接口保留简单关键字契约
2. 内部查询接口支持服务名、级别、状态码、traceId、时间窗组合检索
3. 内部查询接口默认附带时间窗口，避免无界扫描
4. 指标接口直接基于已落库日志做聚合

导出路径：

1. 公开接口创建异步导出任务
2. 后台调度执行导出
3. 导出前做结果规模校验
4. 写文件前对敏感文本做脱敏
5. 文件通过 `/downloads/**` 暴露
6. 到期后由清理任务删除文件并将任务标记为 `EXPIRED`

### 3.2 控制面

`/api/logs/internal/**` 提供服务内部控制面：

- 组合检索
- 运行态总览
- 告警规则管理
- 手动触发刷盘、告警评估、导出执行、导出清理
- 导出任务列表查询

这部分不影响冻结的公开业务契约，但为联调、运维、验收和后续平台化保留了基础面。

## 4. 可靠性设计

### 4.1 写入可靠性

- 入口只做“受理成功”，不做同步重写库。
- 缓冲队列满时立即失败，避免无限阻塞调用方。
- ClickHouse 写入失败后进入重试仓。
- 同一批日志超过最大重试次数后落本地 `failed-batches` 目录。
- 失败不会拖垮服务主线程，错误摘要进入运行态。

### 4.2 查询可靠性

- 内部查询限制最大时间跨度，避免误操作扫全库。
- 关键词长度、分页大小、日志级别、状态码范围都做前置校验。
- ClickHouse 查询异常时回退到内存后备仓，保证本地环境和测试环境可运行。

### 4.3 导出可靠性

- 导出采用任务模型，不在 HTTP 请求线程内生成文件。
- 结果数超过阈值时任务失败，而不是直接生成不可控大文件。
- 导出文件带有效期，避免长期堆积。
- 到期清理失败会记录到运行态。

## 5. 安全与数据治理

当前实现已落地：

- 导出内容中对手机号和 `token/password/secret` 类文本做脱敏。
- 公开响应继续使用统一 `ApiResponse`，避免接口外层漂移。
- 时间统一按 `Instant`/UTC 语义处理。

当前明确假设：

- 多租户权限过滤、按用户数据范围过滤日志，当前仓库没有统一鉴权上下文，暂不在本服务内伪造实现。
- 原文档中的“解析规则平台”和“采集代理下发”需要独立控制面或独立服务，当前仅在规划中保留，不在本次提交里硬塞进 `log-service`。

## 6. 生命周期治理

已落地治理项：

- ClickHouse 表按天分区
- TTL 由配置控制保留天数
- 导出文件按 TTL 清理
- 写入失败批次持久化归档

后续建议补强：

- 失败批次回放工具
- 历史分区容量观测
- 导出审计记录
- 定期索引体检或物化视图优化

## 7. 与原文差距的分阶段规划

### Phase 2A：本次已落地

- 异步缓冲写入
- ClickHouse 适配层
- 重试仓与失败批次归档
- 组合检索与时间窗护栏
- 指标聚合
- 告警规则、告警事件、通知通道
- 异步导出、脱敏、下载映射、过期清理
- 控制面运行态与任务管理

### Phase 2B：建议下一步落地

- 告警去重窗口和抑制策略
- 失败批次回放任务
- 查询性能指标持久化
- 导出任务状态分页检索
- 更细粒度的敏感字段规则
- 基于服务维度的查询限流

### Phase 3：平台级能力

- 日志采集端任务编排
- 日志解析规则管理
- 资源指标采集
- 多租户与权限数据域
- 告警通知记录中心
- 正则检索沙箱与复杂查询审计

## 8. 验收口径

本轮完成后，认为“与原文一致且可落地”的标准是：

- 核心业务链路可以真实跑通，而不是只返回假数据
- 关键生产护栏已经体现在代码和测试里
- 原文中不适合当前仓库一次性实现的部分，有明确纠偏和阶段规划
- 公开契约稳定，内部控制面可支持运维和后续扩展

## 9. 结论

当前仓库中的 `log-service` 不再只是演示骨架，而是已经具备生产化演进方向的单服务落地基础。它没有假装一次性完成整个平台，但已经把原文中最关键、最容易在真实上线时出问题的环节补成了可运行、可测试、可扩展的实现，这就是本次 P2 的正确落地边界。
